제품 링크 하나만 던져서 15초짜리 광고를 만들어보려고 했습니다. Claude + Higgsfield(Seedance 2.0) 한 채팅 안에서 기획, 콘티, 영상까지 전부요.

그런데 렌더를 돌릴 때마다 콘티에서 벗어났습니다. 컷마다 유리잔이 다른 잔이 되고, 사람 얼굴도 바뀌고, 무엇보다 Seedance가 립스틱 디자인이랑 열림/닫힘 상태를 계속 틀렸어요. (학습 데이터에 그 제품이 있었을 리가 없죠.) 그리고 "유리잔을 똑바로 들고 입술을 대는" 포즈는 아무리 고쳐도 마시는 포즈로 나왔습니다. (아마 학습 데이터에는 마시는 영상이 산처럼 많았을 겁니다.)

## 원인 분석

비디오 모델에게 여러 컷으로 된 광고를 부탁한다는 건, 사실 서로 다른 종류의 정보를 한꺼번에 던진다는 뜻입니다.

- **구조**: 무엇이 어떤 순서로 일어나는가
- **외형**: 사람, 제품, 소품이 어떻게 생겼는가
- **계약**: 정확한 첫 프레임과 마지막 프레임
- **움직임**: 어떻게 움직이는가

이걸 전부 텍스트 프롬프트 하나에 밀어 넣으면(제가 처음에 그렇게 했습니다), 채널끼리 서로 치고받다가 모델이 슬슬 딴 길로 갑니다. 처음 원샷 렌더는 600토큰쯤 되는 프롬프트에서 나왔는데, 핵심 컷이 콘티를 벗어났고 원래 없던 주근깨까지 얼굴에 잔뜩 생겼습니다. 90점까지는 정말 빠른속도로 도달하는데 여기서부턴 헬 난이도로 가더군요. 점점 프롬프트와 단계, 그리고 Reference 이미지가 늘어났는데 99점 이상을 줄 수 있는 결과물이 나오지 않았습니다.

먼저 인물의 피부톤이 변했는데요. 제가 프롬프트에 `natural moles, visible pores` 같은 피부 묘사를 넣었는데, **텍스트가 레퍼런스 이미지를 이겨버린** 거예요. 실제로는 외형이 이미지에서 와야 하는데, 텍스트로 다시 설명하니 그 특징이 랜덤하게 더해졌습니다.

> 레퍼런스가 보여주는 걸 텍스트로 다시 묘사하지 말자

## 최종: 관심사의 분리(Separation of Concerns)

핵심은 정보의 종류마다 가장 잘 담기는 형식이 존재합니다. 프로그래밍에서 말하는 관심사의 분리(Separation of Concerns)와 똑같아요. 하나의 책임은 하나한테만 맡깁니다. 위에 예시처럼 인물의 피부에 대해서는 레퍼런스 이미지가 외형을 담당해야지 텍스트나 콘티 같은 다른 요소로 묘사하지 않는 겁니다. SOLID의 SRP(단일 책임 원칙)가 클래스에 하나의 책임만 주듯이, 여기서도 각 레퍼런스는 딱 하나만 맡습니다. 구조는 구조만, 외형은 외형만요.

| 요소   | 담는 것                           | 형식                       |
| ------ | --------------------------------- | -------------------------- |
| 콘티   | 컷 순서, 프레이밍, 타이밍, 카메라 | 주석 달린 스케치 콘티      |
| 이미지 | 사람, 제품, 소품의 생김새         | 라벨이 붙은 실사 사진 시트 |
| 앵커   | 첫 프레임과 마지막 프레임         | 시작/끝 키프레임 앵커      |
| 움직임 | 손동작, 모션                      | 공식 사용 영상(`--video`)  |
| 텍스트 | 액션, 카메라, 타이밍만            | 프롬프트(외형 재묘사 없음) |

말은 간단한데, 여기까지 오는데 이틀은 족히 걸린 것 같습니다.

## 최최종: 콘티는 스케치로

처음에는 실사에 가까운 콘티 시트를 넣었습니다. 사람이 보기엔 그럴듯하지만, 비디오 모델한테는 그게 **두 번째 외형 신호**처럼 작동해서 사고가 났습니다. 콘티에 그려진 얼굴이 캐릭터 레퍼런스를 오염시켰고("유사하지만 다른사람" 등장), 소품 상태도 결과물에서 변했습니다. 전체적인 스토리라인만 잡으려고 했던 콘티인데, 비디오 모델이 이걸 이미지로 받아들이면서 여러모로 문제가 생겼습니다.

이 문제를 해결하기 위해 많은 블로그 포스팅과 논문도 살펴봤는데요. 제 결론은 연필 스케치로 바꾸는 것이였습니다. 스케치에는 외형 정보가 없으니, 모델이 그림을 재현할 대상으로 읽지 않고 **주석을 지시사항으로 읽습니다**.

![연필 스케치 콘티 - 인물은 얼굴 없는 제스처 포즈이고, 각 패널에 타임코드, 카메라 메모, 상태 라벨이 붙어 있음](./2026-07-20/storyboard-sketch.png)

이건 연구 쪽에서도 근거가 있습니다. [DrawVideo](https://arxiv.org/html/2605.23508v1)는 각 샷의 조건을 흑백 스케치(구도/레이아웃) / 외형 프롬프트 / 모션 프롬프트로 나누는 걸 설계의 전제로 삼습니다. 스케치는 처음부터 외형의 기준이 아니었던 거죠. 한 [실전 워크플로우 글](https://oimi.ai/en/blog/gpt-image-2-seedance-2-workflow)에서도 단일 이미지 방식은 "랜덤 점프 컷" 때문에 다섯 번 넘게 실패했지만, 타임라인 주석이 있는 스토리보드는 첫 시도에 성공했다고 합니다. 전통 애니메이션 보드가 썸네일 단계에서 일부러 디테일을 포기하고 카메라와 구도만 고정하는 것도 같은 이유입니다.

## 최최최종: 실물이 있으면 실물

제품 시트 생성도 계속 저를 괴롭혔습니다. 립의 캡 방향, 어플리케이터 팁 모양, 양쪽 끝의 색이 계속 흔들렸어요. 왜 그럴까? 생각해보면 이유는 명확합니다. 모델이 이런 데이터를 학습한 적이 없죠. (신제품이니까요.) 제품의 상세 페이지의 여러 이미지를 주고 생성하고 수정해도 크레딧은 더 쓰는데 실제 제품에서는 더 멀어졌습니다. 결국 제가 직접 찍은 실사 사진으로 `master-sheet`를 만들었습니다. 물론 AI 가 뭘 구별 못 하는지 알고 그걸 정확하게 집어주는 데이터가 필요합니다. 그래서 저는 양쪽 립이 닫히고 열려있는 모습을 정확히 캐치할 수 있는 시트를 추가했습니다.

![실사 사진으로 조립한 제품 마스터 시트 - CLOSED / PINK=LIP / SILVER=FIXER 라벨](./2026-07-20/twinlip-sheet.png)

상태나 역할이 달라지는 요소(유리잔의 CLEAN/MARKED, 제품의 LIP/FIXER)는 라벨이 붙은 상태 시트 하나로 만들고, 프롬프트에서 "몇 번 샷에 어떤 상태인지"를 고정했습니다. 이제 리뷰는 그림과 라벨이 서로 모순되지 않는지만 확인하면 됩니다.

## 스킬로 접기

원칙이 맞는다는 걸 확인한 뒤에는 Claude Code 스킬로 저장했습니다.

- **`storyboard` 병합**: 형식을 하나의 스케치 폼으로 고정했습니다.
- **`master-sheet`에 세 가지 추가**: (1) 확정된 콘티 표에서 화면에 등장하는 모든 요소를 뽑는 게이트(컵과 소품도 여기서 결정), (2) 실물 우선(실물이 있으면 생성하지 말고 합성), (3) 정밀 정렬 리뷰. 전체 파이프라인에서 픽셀 단위로 보는 유일한 단계입니다.
- **리뷰 게이트 이동**: 스케치는 썸네일 수준이니 표와 대조만 합니다(라벨, 타임코드, 모순). 자국 위치나 소품 디자인 같은 픽셀 체크는 master-sheet 단계로 옮겼습니다. 제가 만든 출력에 스킬 자체 체크리스트를 돌려보니, 제가 놓친 오류를 잡아냈습니다.

나머지도 스킬 문장으로 적어뒀습니다. 프롬프트에서 외형 단어 빼기, 실사 사진 합성하기, 중복 과금 피하기(네트워크 호출이 끊겨도 서버에는 제출이 살아 있을 수 있으니 다시 제출하기 전에 확인하기). 다음 캠페인에서 같은 실수를 반복하지 않으려고요.

## 정리

- 여러 컷짜리 영상을 텍스트 프롬프트 하나로 요청하면 구조, 외형, 움직임이 한 채널 안에서 경쟁하다가 결국 흔들립니다.
- 채널을 나누세요. **구조 = 스케치 콘티 / 외형 = 실사 사진 시트 / 계약 = 시작·끝 앵커 / 움직임 = 비디오 레퍼런스 / 텍스트 = 액션만.**
- 실사풍 콘티는 두 번째 외형 신호가 되어 사고를 냅니다. 스케치에는 외형이 없으니 주석이 지시사항으로 읽힙니다.
- 레퍼런스가 보여주는 걸 텍스트로 다시 묘사하면 그 특징이 증폭됩니다("주근깨 사건").
- 실물이 있으면 생성하지 말고 합성하세요. 더 싸고 정확합니다.
- 원칙이 맞으면 스킬로 접어 넣으세요. 다음 세션은 같은 실수를 반복하지 않습니다.
